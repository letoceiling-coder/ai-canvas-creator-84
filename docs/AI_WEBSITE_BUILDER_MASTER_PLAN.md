# AI Website Builder — Master Plan (SSOT v4.1)

**Цель:** продукт уровня Lovable / Framer AI / Devin-lite — многоэтапный pipeline с **внешним контекстом и инструментами** (не «генерация из головы»), устойчивость к сбоям, контроль токенов и моделей, объяснимость, production-ready выход.

**Принципы:** zero-template (seed + DNA + **template detection**), видимость процесса (**streaming**), контроль качества (**quality scores + real QA**), human-in-the-loop, **конфигурируемый pipeline** под тарифы SaaS, **осмысленные tool calls** (по необходимости, с бюджетом), **оценка и отбор результатов tools** (feedback, ranking, кэш).

**Версии документа:** v2 — память, агенты, QA, экспорт; **v3** — model router, compression, retry, quality score, промпты, параллелизм, isolation, semantic validation, self-correction, template guard, pipelineConfig, streaming UX; **v4** — Tool Layer (Context7 + MCP), контракт, стратегия вызовов; **v4.1 (настоящий документ)** — **tool intelligence** (feedback), **ranking**, **политика кэша**, **tool-aware prompts**, **обработка сбоев tools**, **multi-tool composition**.

**Навигация:** архитектура и Tool Layer — **§1** (включая **§1.11–§1.16** — v4.1); память — **§2**; агенты классика — **§3–§15**; оркестратор — **§16**; фазы — **§17**; правила — **§18**; эволюция — **§19**; **§20** — детали v3; **§21** — журнал реализации в репозитории.

## 1. Архитектура

### 1.1 Стек целиком (v4)

```
UI (Canvas + Chat + streaming: токены / % / активный агент / tool calls)
  ↓
Orchestrator (pipelineConfig · projectMemory · compression · retry)
  ↓
Agents (Intent · Planner · Architect · Engineer · Critic · QA · Reviewer)
  ↓
TOOL LAYER  ←  шина: бюджет · **кэш (§1.13)** · **ranking (§1.12)** · **feedback (§1.11)** · **failover (§1.15)** · **compose (§1.16)**
  │   ├─ Context7 : query → extract → **rank** → compress → inject
  │   ├─ MCP: Web Search
  │   ├─ MCP: UI Inspiration
  │   ├─ MCP: Image
  │   └─ MCP: Data
  ↓
Model Router → LLM (**tool-aware prompts §1.14** + урезанный контекст только после **top-K ranking**)
  ↓
parallel: Real QA ∥ Critic (по конфигу) · безопасный параллелизм image/search
  ↓
Quality Score · semantic validation · template check
  ↓
Reviewer (+ self-correction без полного restart)
  ↓
Skills (релевантные chunks) · Prompt Registry
  ↓
Renderer · Export / Deploy · Metrics
```

### 1.2 Зачем слой между агентами и LLM

| Проблема без Tool Layer | Что даёт слой |
|-------------------------|---------------|
| Изолированные агенты, нет внешнего контекста | Доступ к докам, рынку, паттернам, данным по политике |
| Устаревшие практики в коде | Context7: актуальные сниппеты и guidelines (сжатые) |
| «Похоже на AI», не на production | Референсы и data приближают к реальным сайтам |
| Random Unsplash | Image tool: намерение + бренд + секция → релевантный визуал |

**Инвариант v4:** оркестратор не «обходит» Tool Layer там, где шаг помечен как **tool-capable**; агент либо явно вызывает tool, либо фиксирует в `decisionLog` обоснованный отказ («запрос локальный, внешний контекст не требуется») — см. **§1.8**.

### 1.3 Tool Layer — архитектура

#### 1.3.1 Определение

**Tool Layer** — управляемая подсистема между **агентами** и **LLM**, предоставляющая унифицированный доступ к внешним возможностям: документация (Context7), поиск, UI-референсы, изображения по смыслу, структурированные данные. Слой отвечает за **маршрутизацию запросов**, **нормализацию ответов**, **бюджет токенов**, **кэш/дедупликацию** и **аудит** (что вызывалось, зачем).

#### 1.3.2 Как работает

1. Агент (или оркестратор по политике шага) формирует **намерение вызова**: тип tool, запрос, ограничения (язык, вертикаль, глубина).
2. **Проверка кэша** (**§1.13**): при попадании — пропуск провайдера, всё равно пройти **ranking** под текущий шаг и **feedback** (можно пометить «из кэша»).
3. Tool Layer проверяет **pipelineConfig**, **rate limits**, **политику данных** (PII, лицензии изображений).
4. Выполняется провайдер (Context7 / MCP); при сбое — **§1.15**.
5. **Extract → нормализация items[]** под **§1.4**.
6. **Ranking** (**§1.12**): сортировка по релевантности, свежести, доверию источнику; отбор **top-K** под бюджет токенов.
7. **Compress** остатка и **inject** в промпт только для отобранных элементов.
8. **Tool feedback** (**§1.11**): сразу после вызова — оценка полезности/качества; **usedInFinal** уточняется постфактум при фиксации артефакта.
9. В `projectMemory` — **typed attachments** + запись в след аудита (**§2.4**).

#### 1.3.3 Как вызывается агентами

| Режим | Описание |
|-------|----------|
| **Agent-driven** | LLM с function/tool calling возвращает явный вызов; оркестратор исполняет и подмешивает результат в следующий раунд. |
| **Orchestrator-mandated** | На шаге задано правило: «Planner обязан search при новой вертикали» — минимум один вызов или записанное исключение. |
| **Hybrid (рекомендуется v4)** | Обязательные минимумы по тарифу + модель решает дополнительные вызовы в рамках бюджета. |

#### 1.3.4 Интеграция в pipeline

- Tool Layer **не заменяет** `projectMemory` и **не ломает** context isolation (**§20.7**): результаты попадают в память как **typed attachments** (например `toolResults.planner.search`) и в **view** следующих агентов только в сжатом виде.
- Join-point: после каждого tool-блока оркестратор **обновляет decisionLog** (какой tool, зачем, сколько токенов ушло на инъект) и пишет **toolFeedback** (**§1.11**) + краткий итог **ranking** (сколько отброшено, top-K).
- Параллелизм: независимые tools (например UI inspiration + узкий search) — **parallel с лимитом воркеров**, см. **§20.6**.

### 1.4 Tool Interface — единый контракт

Все провайдеры (Context7, MCP) отдают наружу **один и тот же логический набор каналов** и нормализованный результат — см. таблицу и поля ниже.

#### 1.4.1 Логические инструменты

| Ключ | Назначение | Вход (логически) | Выход (логически) | Ограничения |
|------|------------|------------------|-------------------|-------------|
| **context** | Документация, API, best practice | Запрос + область (стек, версия библиотеки) | Список **chunks**: заголовок, краткий текст, источник, актуальность | Жёсткий **token budget** на инъект; без сырого dump |
| **search** | Веб / рынок / конкуренты | Запрос + язык + глубина (top-K) | Нормализованные **выдержки** + URL/title + дата если есть | Кэш; фильтр мусора; лимит вызовов на сессию |
| **ui** | Паттерны интерфейсов | Тип страницы, индустрия, желаемый паттерн | Структурное описание: блоки, иерархия, типичные секции | Только **идеи структуры**, не копипаст чужого UI |
| **image** | Визуалы по смыслу | Бриф секции + Style DNA + ограничения бренда | URL или asset-id + **alt/reasoning** кратко | Политика лицензий; запрет random unrelated stock |
| **data** | Факты, цифры, копирайт-опоры | Тема + юрисдикция/язык + тип данных | Структурированные поля (таблица фактов) | Источник обязателен; при отсутствии — «unknown», не выдумывать |

#### 1.4.2 Формат ответа инструмента (норма)

- **summary** — 1–3 предложения «что важно для агента».
- **items[]** — унифицированные элементы (chunk, result, pattern, image ref, fact).
- **metadata** — источник, timestamp, confidence, **estimatedInjectTokens**.
- **redacted** — флаг, если контент обрезан политикой.

#### 1.4.3 Token budget

- Глобальный лимит на инъект из всех tools в один LLM-вызов (согласовать с **§20.2**).
- При переполнении: сначала **ranking** (**§1.12**), затем отрезание хвоста; политика приоритетов (например context для Engineer выше сырого search) — как в **§1.9**.

### 1.5 Context7 (Docs Provider)

#### 1.5.1 Роль

Поставщик **актуальной документации и практик** для стека генерации (React, a11y, performance, а при необходимости — бэкенд шаблонов). Заменяет «знания из головы» на **верифицируемые выдержки**.

#### 1.5.2 Поток: query → extract → compress → inject

| Фаза | Действие |
|------|----------|
| **query** | Формируется из намерения Engineer/Architect: библиотека, тема, версия, тип задачи (layout, forms, a11y). |
| **extract** | Получение сырья из Context7 (или совместимого индекса доков). |
| **rank** | Ранжирование chunks по **§1.12** (relevance, freshness, trust); отбор кандидатов в **top-K** до сжатия. |
| **compress** | Извлечение лучших фрагментов после rank; удаление воды; ссылки на первоисточник. |
| **inject** | В промпт попадает **только top-K** в рамках бюджета (**§1.4.3**). |

#### 1.5.3 Фильтрация

- По **релевантности** к текущему шагу (embedding / keyword route).
- По **возрасту** документа (предпочтение свежим guideline при конфликте).
- По **лицензии** контента (только разрешённые источники).

#### 1.5.4 Инвариант

**Запрещено** вставлять в LLM целые статьи целиком. Разрешено: **сжатые**, **цитируемые** фрагменты + указание источника в памяти для аудита.

### 1.6 MCP Tools — обязательный набор (минимум v4)

#### 1.6.1 Image Tool

- Вход: семантика секции, намерение сайта, Style DNA, ограничения пользователя (стиль фото, запрет лиц, бренд-палитра).
- Выход: изображение **осмысленное** для блока (hero, feature, gallery), с **обоснованием выбора** и alt-текстом.
- **Не** допускается по умолчанию: случайный Unsplash без связи с брифом (legacy-путь — только fallback при отключённом tool или аварии провайдера, с логированием).

#### 1.6.2 Web Search Tool

- Поиск **реальных сайтов**, продуктовых формулировок, трендов по индустрии, актуальных формулировок CTA в вертикали.
- Результат: выдержки + источники; **не** копировать тексты дословно в коммерческий копирайт без переработки (политика + Critic).

#### 1.6.3 UI Inspiration Tool

- Паттерны: навигация, hero-типы, сетки pricing, social proof, footer-структуры.
- Выход: **структурное** описание (иерархия блоков, названия секций, связи), без переноса визуального кода с чужих сайтов.

#### 1.6.4 Data Tool

- Факты, типовые цифры по индустрии (где этично и без выдумывания), подсказки для копирайта **с опорой на источник**.
- Если факт не найден — явный **data gap** в памяти; Engineer не заполняет вымышленными метриками.

#### 1.6.5 Расширения

Дополнительные MCP (брендбук с Figma, CMS, analytics) подключаются тем же контрактом **§1.4**, с отдельными политиками тарифа.

### 1.7 Агенты + использование tools

| Агент | Tools (минимум / типично) | Зачем |
|-------|--------------------------|-------|
| **Intent** | Опционально `search` при неоднозначной нише | Уточнение вертикали и языка рынка |
| **Planner** | **`search`** — референсы конкурентов/топовых лендингов; **Data** при необходимости фактов для УТП | План опирается на реальные рынковые паттерны |
| **Architect** | **`ui`** — паттерны layout и блоков; опционально `search` для структуры отрасли | Архитектура не «из головы» |
| **Engineer** | **`context` (Context7)** — практики кода/a11y/perf; **`image`** для секций с визуалом; при необходимости узкий `search` для копирайта | Production-ready код и осмысленные медиа |
| **Critic** | **`search`** — сравнение с публичными эталонами / формулировками (в рамках этики) | Оценка «уровня рынка», не абстрактная |
| **QA** | **Data** (проверка заявленных фактов против сохранённых источников) + **rules** (автоматика из §8) | Не только линтер, но и согласованность с данными |
| **Reviewer** | Опционально `context` — финальный проход по стилю кода/текста | Полировка с опорой на гайды |

### 1.8 Tool call strategy

#### 1.8.1 Принцип

**Агент сам инициирует tool только при необходимости** — но оркестратор применяет **минимальные обязательные вызовы** по тарифу и типу задачи (см. hybrid в **§1.3.3**).

#### 1.8.2 Когда вызывать

| Условие | Действие |
|---------|----------|
| Новая вертикаль / незнакомая ниша | Planner: `search` (+ при желании Data) |
| Сложный или новый стек в генерации | Engineer: `context` (Context7) |
| Секция требует визуала | Engineer или отдельный шаг: `image` |
| Критик фиксирует «generic / не рынок» | Повтор Critic или Architect с усиленным `search` / `ui` |
| Локальная правка одной секции | Tools **можно пропустить**, если в памяти уже есть достаточный контекст (кэш сессии) |

#### 1.8.3 Когда не вызывать

- Повторный вызов с тем же **каноническим запросом** в одной сессии — брать **кэш** Tool Layer.
- Запрос пользователя **чисто косметический** (цвет, текст одной кнопки) без смены нише/структуры.
- Достигнут **лимит вызовов** тарифа — деградация с явным уведомлением в UI и записью в `decisionLog`.

#### 1.8.4 Избежание лишних вызовов

- **Дедупликация** по хэшу (**§1.13**, ключ кэша).
- **Batch**: один search с несколькими подвопросами вместо серии мелких.
- **Tiered depth**: сначала короткий поиск / узкий context; расширять только если qualityScore низкий.

### 1.9 Context management (tool outputs + память)

- Результаты tools **не** подмешиваются в «простыню» `projectMemory`: только **нормализованные записи** + **сжатый summary** для views агентов (**§20.7**).
- **Сжатие** на границе Tool Layer согласовано с **§20.2**: общий потолок токенов на шаг делится между памятью и инъектом из tools.
- **Релевантность:** в промпт попадают chunks с наивысшим score по текущему шагу; остальное — только в архив памяти для аудита.
- **Прозрачность:** в streaming UX (**§20.12**) опционально отображать **имя tool** и **краткий итог** (без утечки больших тел ответов).

### 1.11 Tool Intelligence (feedback)

**Проблема:** модель вызывает tool, получает шум, перегружает контекст или нерелевант — без обратной связи это повторяется.

**После каждого завершённого вызова** (успех или деградация после **§1.15**) в память пишется запись минимум с полями:

| Поле | Смысл |
|------|--------|
| **useful** | Бинарная оценка: результат реально сдвинул рассуждение / структуру / код (эвристика + при возможности мини-классификатор / правило). |
| **quality** | 0–100: связность с запросом, плотность фактов, отсутствие мусора, согласованность источников. |
| **usedInFinal** | Обновляется после сборки черновика / финала: попали ли item’ы этого вызова в итоговый артефакт (трассировка по id chunk / секции). |

**Использование сигнала:** оффлайн — калибровка порогов, промптов, **tool-aware** шаблонов (**§1.14**); онлайн — правила «при серии низкого **quality** для `search` — упростить запрос / сменить глубину»; в метриках SaaS — доля бесполезных tool calls, средний **quality** по каналу.

### 1.12 Tool result ranking

**Инвариант:** в промпт **никогда** не уходит сырой полный список результатов провайдера — только после **rank + top-K**.

**Ось ранжирования (минимум):**

| Сигнал | Зачем |
|--------|--------|
| **Relevance** | Близость к запросу шага и к `userIntent` / `plan` (embedding + ключевые требования). |
| **Freshness** | Для search/docs: дата/версия; устаревшее опускается, кроме явных «классических» источников в whitelist. |
| **Trust** | Репутация домена / тип источника (дока библиотеки vs форум), штраф за низкое качество источника. |

**Процедура:** нормализовать items → вычислить скор → сортировка → **top-K** где K и пороги отсечения задаются `pipelineConfig` и оставшимся token budget. Низкоранжированное остаётся только в архиве памяти для аудита.

### 1.13 Tool caching (стратегия)

**Цель:** скорость, стабильность и снижение стоимости inference без ущерба для релевантности.

| Параметр | Рекомендация |
|----------|----------------|
| **Ключ** | Хэш от **нормализованного запроса** + **роль агента** + **intentType** (и при необходимости канал tool); опционально версия «корпуса» docs для Context7. |
| **TTL** | По типу: search / новости — короткий (например до 1 ч); статичные guideline — дольше; конфигурируемо по тарифу. |
| **reuseAcrossSteps** | **true** по умолчанию внутри сессии: один и тот же кэшит может кормить Planner и позже Critic, если ключ совпал и TTL валиден. |
| **Инвалидация** | Смена **userIntent**, жёсткий **partial regen** другого scope, смена **styleDNA/plan** по HITL — ключи помечаются устаревшими или инкремент **sessionGenerationEpoch**. |

Из кэша отдаётся нормализованный ответ; **ranking** всё равно выполняется заново относительно **текущего шага** (порядок top-K может отличаться).

### 1.14 Tool-aware prompts

**Проблема:** агент «видит» tools только через API, а системная инструкция не знакома с дисциплиной вызова — растёт число лишних или нецелевых обращений.

**Правило v4.1:** в **системном слое** каждого tool-capable агента явно перечислены доступные каналы **search · context · ui · image · data** (только фактически разрешённые `pipelineConfig`) и дисциплина:

- Вызывай инструмент **только если** не хватает фактов, актуальности, структурных паттернов или нужна верифицируемая дока.
- **Не** вызывай при локальной косметической правке при warm-кэше и полноте контекста (**§1.8**).
- После получения результата **полагайся на отранжированный top-K**, не дублируй запрос без новой гипотезы.

Версии этих фрагментов — в **prompt registry** вместе с основным текстом роли (**§20.5**).

---

### 1.15 Tool failure handling

**Отдельно от LLM retry (**§20.3**):** сетевой сбой, таймаут MCP, пустой ответ, rate limit провайдера.

**Цепочка деградации (по политике):**

1. **Повтор** с упрощённым / укороченным запросом (меньше top-K на стороне провайдера).
2. **Альтернативный канал**: другой tool с пересекающейся целью (например при падении Data — узкий search с жёстким whitelist).
3. **Без tool** с обязательной записью в `decisionLog` + маркер в streaming UX; downstream агенты не предполагают наличие внешних данных.
4. Для Context7: fallback на **кэшированный** срез или урезанный **skills**-корпус продуктовой команды (если разрешено политикой **§1.13**).

Повторы инструмента лимитировать отдельно от повторов LLM; circuit breaker на провайдера при серии ошибок.

---

### 1.16 Multi-tool composition

**Уровень выше «один вызов — один ответ»:** связка даёт **согласованный** бриф для следующего агента.

**Примеры рецептов (оркестратор или политика шага):**

| Цепочка | Назначение |
|---------|------------|
| **search → ui** | Сайты/формулировки рынка → структурные паттерны блоков под ту же вертикаль. |
| **search → data** | Сначала контекст нише, затем точечные факты с источниками. |
| **ui → context** | Выбранные паттерны → как технически реализовать (docs, a11y). |
| **search + context** (parallel → merge) | Референсы и практики одновременно → единый **composed brief** перед Engineer. |

**Правила merge:** conflict resolution (конфликт доки и search — приоритет доке с высоким **trust**); общий **token ceiling** на composed пакет; одна запись **toolFeedback** на **этап compose** + при необходимости по подвызовам.

## 2. Project Memory (критично)

Агенты **не stateless**: каждый шаг читает и дополняет единый контекст проекта.

### 2.1 Структура `projectMemory`

```ts
projectMemory = {
  sessionId: string
  userIntent: string              // исходный промпт + уточнения
  intentType: IntentType         // см. §5
  styleDNA: StyleDNA             // см. §10
  designSeed: DesignSeed         // layout, spacing, typography, colorSystem, animationStyle
  plan: PlannerOutput | null
  architecture: ArchitectOutput | null
  codeRef: CodeBundleRef | null  // указатели на версии кода/дерева секций
  decisionLog: DecisionLogEntry[]
  constraints: Constraint[]       // бренд, запреты, юридические, язык
  userEdits: UserEdit[]           // inline / структура / правки плана
  metrics: SessionMetrics         // см. §13
}
```

### 2.2 Правило

- **Каждый агент** в вызов LLM получает не сырой лог, а **view памяти** (см. **§20.7** Context isolation) после **сжатия** (§20.2).
- После шага агент **обновляет** память через **строго типизированный патч** (merge), валидируемый Zod.
- Orchestrator хранит версии памяти (для отката и partial regen).

### 2.3 Связь с Token Control (v3)

Детали в **§20.2**: лимит токенов на контекст, summarize старых чанков, decisionLog в краткой форме, skills — только релевантные фрагменты.

### 2.4 Следы инструментов (v4.1)

Память дополняется **аудируемым** слоем вызовов Tool Layer (не смешивать с «простынёй» промпта):

- **toolInvocations[]**: канал, нормализованный запрос, ключ кэша, флаг попадания в кэш, итог **ranking** (top-K), **оценка toolFeedback** (**useful**, **quality** 0–100), ссылка на сырой архив при необходимости.
- **usedInFinal** проставляется **постфактум** при фиксации артефакта (сопоставление chunk id / секций с итоговым сайтом) — для оффлайн-аналитики и калибровки **§1.11**.
- Результаты **multi-tool composition** (**§1.16**) хранятся как одна запись compose + дочерние ссылки на подвызовы.

## 3. Decision Log

Массив осознанных решений для объяснимости и стабильности стиля.

```ts
type DecisionLogEntry = {
  id: string
  agent: "planner" | "architect" | "engineer" | "reviewer" | "critic" | "system"
  summary: string                 // коротко для UI
  detail?: string                 // опционально для раскрытия
  relatedIds?: string[]           // секции, компоненты, токены дизайна
  createdAt: string
}

// Примеры:
// "Выбран минимализм для повышения конверсии"
// "CTA размещён выше fold"
// "Dark theme для премиум-восприятия"
```

**UI:** в чате отдельная сворачиваемая лента «Почему так» или встроенные сноски к этапам.

## 4. Multi-Iteration Design Loop

Один проход «сгенерировал → готово» недостаточен для топового UI.

### 4.1 Параметры

- `designLoopMin = 2`, `designLoopMax = 3` (настраиваемо по тарифу/качеству).
- На каждой итерации: **внутренний** цикл «сгенерировать вариант → анализ (Critic + проверки) → улучшить».

### 4.2 Встраивание в pipeline

После **Engineer** (или параллельно секциям):

1. `draft = engineer(architecture, memory)`
2. `critique = critic(draft, memory, styleDNA)`
3. `improved = engineerRefine(draft, critique, memory)` (только дифф/секции с низким score)
4. Повтор до лимита итераций или пока **qualityScore** (§20.4) не достигнет порога (например ≥ 80 по взвешенной сумме).

**QA** и **validateDesignSystem** (§7) входят в критерий «принять итерацию». **Semantic validation** (§20.7) — дополнительный gate.

## 5. Intent Classifier (до Planner)

**Задача:** классифицировать запрос до тяжёлого планирования.

```ts
type IntentType =
  | "landing"
  | "ecommerce"
  | "saas"
  | "portfolio"
  | "blog"
  | "docs"
  | "other"

type IntentResult = {
  intent: IntentType
  confidence: number
  hints: string[]        // для Planner: «B2B», «лидоген», и т.д.
}
```

**Реализация:** лёгкая модель / эвристики + один короткий LLM-вызов при низкой уверенности.

**Эффект:** Planner и Architect получают `intent` в `projectMemory` — меньше «угадываний» структуры.

## 6. Style DNA

Убивает ощущение шаблона: **общий «характер»** сайта, согласованный между агентами.

```ts
type StyleDNA = {
  vibe: "futuristic" | "luxury" | "brutal" | "playful" | "corporate" | "editorial" | ...
  density: "airy" | "balanced" | "compact"
  motion: "smooth" | "snappy" | "minimal"
  contrast: "soft" | "medium" | "high"
  // опционально: referenceMood, forbiddenPatterns[]
}
```

- Генерируется или уточняется после **Intent + первого уточнения пользователя**.
- **Передаётся во все промпты** + лежит в `projectMemory`.
- Пользователь может **редактировать** Style DNA в HITL (§9) до фиксации плана.

## 7. Design System Validation

После **Architect** (и после правок Engineer):

```ts
validateDesignSystem(ds: DesignSystem): ValidationReport {
  contrast: ...          // WCAG по парам foreground/background
  spacing: ...           // кратность базовому шагу, отсутствие «ломаных» значений
  typography: ...        // масштаб типографики (модуль/ступени), max длина строки
  tokens: ...            // дубликаты, конфликт имён
}
```

- **Не проходить** дальше или **авто-патч** Architect/Engineer при `severity: error`.
- Результаты попадают в **QAReport** и при необходимости в **decisionLog**.

## 9. Human-in-the-Loop (усиление)

Помимо **confirm plan**:

| Действие пользователя           | Эффект |
|--------------------------------|--------|
| Редактирование плана (текст/JSON UI) | Патч `plan` → повтор Architect при необходимости |
| Редактирование структуры (дерево)   | Обновление `architecture` / порядка секций |

## 10. Partial Regeneration

Избегать полной пересборки при локальном запросе.

```ts
regenerate(scope: { type: "section"; sectionId: string } | { type: "component"; id: string })
```

- Orchestrator передаёт в Engineer **только** поддерево + **срез** `projectMemory` (styleDNA, designSeed, соседние секции).
- Остальной сайт **заморожен** до завершения патча.
- После патча: **инкрементальный** QA (секция + регрессия контраста/навигации).

**UI:** «Переделать hero», контекстное меню секции в превью.

## 11. Component Intelligence

Правила домена для осмысленных блоков (не «просто div»).

```ts
componentRules = {
  hero: { mustHave: ["primaryCTA", "headline"], optional: ["socialProof"] }
  pricing: { mustHave: ["plans", "period"], minPlans: 2 }
  footer: { mustHave: ["legalOrCopyright"] }
  // ...
}
```

- **Architect** обязан удовлетворять правилам для выбранного `intent`.
- **QA** проверяет правила программно + Critic комментирует нарушения смысла.

## 12. Self-Critic Agent (Critic)

Отдельная роль **после** черновика и **между** итерациями design loop.

**Задачи Critic:**

- Сравнение результата с `userIntent`, `plan`, `styleDNA`.
- Список конкретных недостатков («слабый CTA», «перегружен fold», «ломается иерархия»).
- **Не** обязан править код — только диагностика для Engineer/Reviewer.

Выход: `CriticReport { scores, findings[], qualityScore? }` → в память и в чат (краткая выжимка). Нормализованный **qualityScore** см. **§20.4**.

## 16. Orchestrator (логика v4 — с Tool Layer)

**Цепочка высокого уровня:** Intent → **Planner (+ search / data при политике)** → HITL plan → **Architect (+ ui, при необходимости search)** → **Engineer (+ context7, image, узкий search/data)** → Design loop **Critic (+ search)** + **QA (+ data сверка + rules)** → **template / semantic gates** → **Reviewer (+ опционально context)** → render/export.

```text
1. intent = classify(userPrompt) → memory  [router.intent; опционально search §1.7]
2. [optional] clarify → memory.constraints / styleDNA
3. plan = planner(memory + toolLayer.search[/data]) → memory.plan + decisionLog  [router.planner]
4. HITL: confirm / edit plan
5. memory.designSeed = seed() (если не зафиксирован пользователем)
6. architecture = architect(memory + toolLayer.ui[/search]) → validateDesignSystem → memory  [router.architect]
7. loop i in 1..pipelineConfig.designIterations:
     memory.toolContext = inject(compressed tool results: context7, images, при необходимости search/data)
     draft = engineer(memory)  [router.engineer; retry+fallback §20.3; обязательно context7 при tool-capable шаге]
     parallel when enabled:
        critique = critic(draft, memory + toolLayer.search)  [router.critic]
        realQaDraft = realQA(draft) + data-facts cross-check vs memory.toolResults
     merge; semanticCheck (§20.7); qualityScore (§20.4)
     if score < threshold: engineerRefine / self-correct (§20.8)
     if qualityThreshold: break
8. issues = остаточные из realQA + llmQA
9. if issues: engineerFix(draft, issues)
10. templateSimilarityCheck (§20.9)
11. final = reviewer(draft, memory) [+ опционально context7 полировка]
12. render + export; metrics (добавить счётчики toolCalls, injectTokens)
```

- На каждом LLM-шаге после tool-инъекта: **modelRouter** (§20.1), **memoryCompression** (§20.2), **context view** (§20.7), **§1.9** для tool outputs.
- **§1.8** — когда tools обязательны / пропуск; **§1.13** — кэш и инвалидация; **§1.12** — **rank** перед inject; **§1.11** — **feedback** после вызова; **§1.15** — сбои провайдеров; **§1.14** — **tool-aware** системные промпты; **§1.16** — compose-цепочки там, где продуктовая политика требует «дизайнерского» уровня.
- **pipelineConfig** расширяется флагами: `enableContext7`, `enableMcpSearch`, `enableMcpUi`, `enableMcpImage`, `enableDataTool`, лимиты вызовов на сессию, **TTL по типам кэша**, **maxToolRetries**, пороги **quality** для повторного запроса (§20.11).
- **Partial regen** и правки пользователя: локальный повтор поддерева с **точечными** tool calls (например только `image` для секции); при invalid output агента — **self-correction** (§20.8) до полного перезапуска.

### Фаза 4 — Sandbox + Deploy

- Sandbox для произвольного кода (если включён); Vercel/Netlify.

## 18. Критические правила (v4.1)

### Запрещено

- Один «большой» запрос без этапов.
- Агенты без **projectMemory** и **decisionLog**.
- Игнорирование **validateDesignSystem** и **real QA**.
- Полная перегенерация без **partial scope** там, где пользователь просит точечное изменение.
- **Генерировать «с нуля» без учёта Tool Layer** на шагах, помеченных как **tool-capable** и включённых в `pipelineConfig` (обход делается только с явной записью в `decisionLog` и по §1.8).
- Подмена **отсутствующих данных** выдуманными фактами вместо **data gap** / повторного **Data tool**.
- **Случайные нерелевантные изображения** как основной режим (**§1.6.1**): без Image tool — только задокументированный fallback.

- **Инъект без ranking** — запрещено: только **top-K** после **relevance + freshness + trust** (**§1.12**).
- **Скрытый сбой tool** (молчаливый пропуск без `decisionLog` / streaming) — запрещено; только явная деградация по **§1.15**.

### Обязательно

- **Использовать tools при необходимости** и по минимальным политикам тарифа (**§1.3.3**, **§1.8**).
- **Context7 / context channel**: сжатие по схеме **query → extract → compress → inject** (**§1.5**), без полных dump’ов в промпт.
- **Проверка через Critic** после черновика; при низком quality — итерация с учётом результатов tools (**§16**).
- **Аудит tool calls** в метриках сессии (число вызовов, injected tokens, отказы провайдера, **средний tool quality**, доля **useful**).
- Соблюдение **token budget** для tool outputs (**§1.4.3**, **§1.9**, **§20.2**).
- **Tool feedback** (**§1.11**) фиксировать после каждого завершённого вызова; **usedInFinal** — при закрытии артефакта (**§2.4**).
- **Tool-aware** фрагменты в системных промптах агентов (**§1.14**) в той же версии, что и основной промпт роли (**§20.5**).
- При отказе провайдера — цепочка **§1.15**, не смешивать с LLM-retry без учёта лимитов **tool retries**.

| Остальное | … | partial regen, HITL… | §20 | §1.3–1.10 + §16–§18 |

## 20. Production-grade слой (детали v3; v4 — §1)

### 20.1 Model Router

Одна модель на всех агентов — снижает качество: **код** и **рассуждение/UX** разные задачи.

```ts
type ModelRouterConfig = {
  intent?: string
  planner: string       // например llama3 — логика, ТЗ, текст
  architect: string     // например qwen2.5-coder — структура + системный выход
  engineer: string      // coder-модель
  critic: string        // llama3 — сравнение с intent, находка слабых мест
  reviewer: string
  qaLlm?: string        // короткий UX-QA
  defaultFallback: string
}
```

**Пример (Ollama):** `planner/critic/reviewer/qa → llama3:latest`, `architect/engineer → qwen2.5-coder:7b`. Конфиг из env / БД, не захардкожен в клиенте.

---

### 20.2 Token control (memory compression)

`projectMemory` со временем разрастается → обрезание контекста и потеря связности. Вводится **явный бюджет**:

```ts
memoryCompression = {
  maxContextTokens: number        // например 6000–12000 на вызов
  strategy: "summarize" | "truncate" | "hybrid"
  keepPinned: ["userIntent", "styleDNA", "designSeed", "intentType"]
  decisionLogMaxEntries: number    // старые → summary bullet
  chatHistory: "summarize_per_turn_block"
  skills: "embeddings_or_keyword_filter"  // только релевантные чанки
}
```

**Правила:** после N шагов старые реплики чата сворачиваются в summary; `decisionLog` хранит полные последние K записей + агрегат старых; в промпт не отдаётся полный сырой код предыдущих версий — только hash + дифф-пойнтеры при необходимости.

---

### 20.3 Failure recovery

```ts
retryStrategy = {
  maxRetries: 2
  backoffMs: number
  onJsonParseError: "retry_stricter_prompt" | "repair_with_llm"
  fallbackModel: string          // например llama3 если qwen упал/timeout
  onTimeout: "retry_once" | "switch_model"
  circuitBreaker: optional       // после серии ошибок — деградация режима
}
```

Примеры: **503 / timeout** → повтор с тем же или fallback-моделью; **невалидный JSON** → повтор с «только JSON, без markdown»; после `maxRetries` — мягкий fail с сохранением артефакта и запрос пользователя «продолжить / уточнить».

---

### 20.4 Quality score system

Единая шкала для gate итераций и для метрик SaaS:

```ts
qualityScore = {
  design: number      // 0–100 визуальная целостность, иерархия
  ux: number         // поток, ясность CTA, cognitive load
  performance: number // оценка Lighthouse perf / веса / LCP proxy
  accessibility: number // a11y score + контраст
  semantic?: number   // см. §20.7
}
// aggregate = weighted(…) → порог например 80 для выхода из design loop
```

**Источники:** Critic (субъективная часть), Lighthouse/ESLint (объективная), `validateDesignSystem`. При **aggregate < threshold** — ещё итерация или эскалация в Reviewer.

---

### 20.5 Prompt versioning

Промпты не «висят безымянно» в коде:

```ts
promptVersion = {
  intent: "v1.0"
  planner: "v1.2"
  architect: "v1.4"
  engineer: "v2.0"
  critic: "v1.1"
  // ...
}
```

Реестр: id версии → текст системного промпта + changelog. Нужно для **обновлений**, **A/B тестов**, **отката** при регрессии качества. В `sessionMetrics` пишется набор версий, использованных в сессии.

---

### 20.6 Parallel execution

Строго последовательный pipeline — медленный. **Безопасный** параллелизм:

| Задача | Параллельно с |
|--------|----------------|
| Real QA (lint, контраст) | Critic LLM |
| Подбор изображений / запросы к Unsplash | Engineer finalize (если нет циклической зависимости) |
| Независимые секции (при зрелой архитектуре) | ограниченный пул воркеров |
| Независимые **MCP-вызовы** (search + ui) до merge в память | **§1.3.4**, лимит воркеров Tool Layer |

Оркестратор ждёт **join** перед merge; `pipelineConfig.enableParallelQA = true` для платных тарифов. Ожидаемое ускорение end-to-end — порядка **30–40%** при правильных join-точках; с Tool Layer — дополнительный выигрыш за счёт **batch + cache** (**§1.8**).

---

### 20.7 Context isolation

Ошибка: все агенты видят один и тот же «простыню» памяти → шум и **утечка ролей** (planner начинает «писать код»).

**Решение:** для каждого агента формируется `agentContextView`:

```ts
agentContext = {
  planner: Pick<Memory, "userIntent" | "intentType" | "constraints" | "styleDNA" | "summarizedChat">
  architect: + "plan" | "decisionLogSummary"
  engineer: + "architecture" | "designSeed" | "componentRules"
  critic: + "draftSnapshot" | "userIntent" | "styleDNA" | "planSummary"
  // без лишних технических деталей чужих этапов
}
```

Полный `projectMemory` остаётся у orchestrator; в LLM уходит только **view** + compressed blocks.

---

### 20.8 Semantic validation

Помимо технического QA — **смысловая** согласованность с целью сайта:

```ts
semanticCheck = {
  heroAlignsWithGoal: boolean      // hero отвечает intent / goals из plan
  ctaCoherent: boolean           // CTA ведёт к заявленной цели (лид, покупка, демо)
  blocksNonContradictory: boolean // нет взаимоисключающих сообщений
  // реализация: отдельный короткий LLM-вызов + чеклист из plan.goals
}
```

Провал → запись в QA issues + триггер refine / Critic с фокусом на semantics. Входит в **qualityScore.semantic** или как gate.

---

### 20.9 Agent self-correction

Не поднимать весь pipeline при локальной ошибке:

```text
if (agentOutput.invalid_schema || !zod.parse(output)) {
  same Agent.fixSelf(lastOutput, errors)  // 1–2 попытки
}
if still invalid → retryStrategy (§20.3)
```

Применимо к Planner/Architect/Engineer при парсинге JSON и к мелким нарушениям правил `componentRules`.

---

### 20.10 Template detector

Zero-template требует **измеримости**:

```ts
detectTemplateSimilarity(artifact): {
  embeddingDistanceToCorpus?: number  // сравнение с банком прошлых генераций
  structuralFingerprint: string       // порядок секций + class histogram
}

if (similarity > threshold || fingerprint in hotlist) {
  regenerateWithNewSeed()
  decisionLog += "Повышена уникальность: новый designSeed / layoutType"
}
```

Корпус «плохих» похожих шаблонов обновлять оффлайн; порог калибровать по A/B.

### 20.11 Configurable pipeline

Жёсткий pipeline не подходит для SaaS (тарифы, фичефлаги):

```ts
pipelineConfig = {
  enableCritic: boolean
  enableSemanticCheck: boolean
  enableTemplateGuard: boolean
  designIterations: number
  fixIterations: number
  enableParallelQA: boolean
  qualityThreshold: number
  models: ModelRouterConfig
  promptVersions: PromptVersionMap
}
```

Профили: **Free** (минимум итераций, без parallel, урезанные или без MCP), **Pro** (полный v3 + **Tool Layer по §1**), **Enterprise** (кастомные модели, пороги, **свои MCP**, расширенные лимиты вызовов).

**Расширения v4 (в ту же конфигурацию продукта):** флаги включения Context7 и каждого MCP-канала; `maxToolCallsPerSession` / по типу tool; глубина search; включение **Orchestrator-mandated** минимумов per тариф (**§1.3.3**).

---

### 20.12 Streaming UX

При SSE/WebSocket не только «этап начался/закончился», но:

- **Токены** ассистента в чат (по желанию пользователя — toggle «подробный поток»).
- **Прогресс** этапа: 0–100% (оценка по подшагам или по объёму стрима).
- **Текущий агент** и **модель** (прозрачность + отладка).
- **Tool Layer:** события `tool_start` / `tool_end` с **кратким итогом** (имя канала, число items, top-K после rank, inject tokens — без тела ответа); опционально `tool_feedback` (агрегированный **quality** шага).
- События: `agent_start`, `token`, `artifact_chunk`, `agent_end`, `quality_score`.
